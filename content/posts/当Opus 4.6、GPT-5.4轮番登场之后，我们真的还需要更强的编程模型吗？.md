这两年的 AI 编程圈，多少有点进入了一种“强模型狂热”。从 Anthropic 的 Sonnet 4 到 Opus 4.6，从 OpenAI 的 GPT-5.1-Codex 到 GPT-5.4，各家的发布叙事几乎一致：更强的 coding、更长的 agentic work、更好的 code review、更可靠的 bug fixing。这套说法当然成立，因为模型越强，的确越能吞掉软件实现中的大量偶然复杂性，样板代码、框架胶水、测试脚手架、常规修补都能更高效地自动完成。但问题也恰恰从这里开始。如果一个编程模型已经足够强，强到能处理大部分实现层面的负担，那么我们继续追求“更强”，到底在追求什么？更进一步说，更强的编程模型，真的一定意味着更好的软件生产吗，还是只意味着更快地收敛到“统计上最合理”的平均答案？

如果把软件开发简化成一次信息传输，这个问题会更清楚一些。假设一个优秀软件最终包含的有效信息总量是定值 N，其中人类显式提供的信息量是 p，模型自动补全的信息量是 c，那么 p + c = N。这个式子虽然不严格，但足够有启发性，因为它揭示了强编程模型到底在做什么：模型越强，人类需要显式给出的信息就越少，模型依据先验自动补全的信息就越多。工程上这很合理，可一旦 p 持续下降，最终软件中越来越多的部分就不再由人明确决定，而是由模型根据训练语料、历史代码、主流框架和行业平均最佳实践去“猜”和“补”。于是关键问题就出现了：当输入约束不足时，模型输出会越来越受统计先验支配；先验越强，输出越稳定；输出越稳定，软件就越容易同质化。所以同质化并不是因为模型太弱，而是因为“欠规格 + 强先验”这个组合成立了。你没把目标、边界、偏好和 tradeoff 说清楚，模型就会用它见过最多、最像标准答案的方案替你填空，而一个足够强的模型，恰恰最擅长把这种填空做得平滑、合理、主流，最后产物看起来都“不差”，却也越来越像别人做过的东西。当程序员欢欣鼓舞的享受一句话生成软件的过程的时候，又有哪个用户会为这种高度同质化的“一句话软件“买单呢？

这也是为什么我觉得，编程模型真正该吃掉的，应该是 偶然信息（accidental information），而不是 本质信息（essential information）。前者是那些不决定产品灵魂、却大量消耗时间和注意力的东西，比如样板代码、普通 CRUD、机械性 bug 修复、常见工程模板；后者才是真正决定一个软件独特性的部分，比如你到底在为谁优化、愿意牺牲什么保留什么、系统更重一致性还是更重响应、哪些边界 case 在你这里是核心而不是噪声、组织流程如何塑造架构选择。过去几十年软件工程的主线，其实一直都是不断吞掉 accidental information：汇编被高级语言吞掉，资源细节被运行时吞掉，基础设施被云平台吞掉，界面和交互的一部分被框架吞掉，今天轮到样板实现和局部推理被模型吞掉。这完全合理，也值得继续推进。但一旦我们开始期待模型替我们做 essential information 的决定，事情就变质了，因为那不再是在消除偶然复杂性，而是在用统计先验取代人的真实意图。

从这个角度看，所谓“更会写代码”的边际收益递减，其实并不难理解。软件生产的成本从来不只在于把代码写出来，它还包括把真正想要什么说清楚、证明产物真的被用户需要、以及在团队和组织中持续协作、审计和演化。强编程模型主要压缩的是实现成本，以及一部分验证和集成成本；可当这部分已经被压缩得差不多之后，真正昂贵的摩擦就会暴露出来：需求说不清，系统真实约束没有被表达，代码库与组织上下文太大，验证成本高昂，你不知道这次改动应不应该信，产物虽然“能跑”，却并不符合组织在权限、流程、责任和合规上的真实边界。也就是说，当 accidental information 被大量自动化之后，软件工程的主瓶颈会从“构造实现”转向“构造约束、验证正确性、整合上下文和管理演化”。在这个阶段，继续单点追求一个更会写代码的模型仍然有价值，但它已经不再是最值得押注的方向。

所以我越来越觉得，下一阶段真正重要的，不是抽象地追求“更强模型”，而是追求更强的软件生产系统。这样的系统首先要更擅长识别**约束不足**，而不是在**约束不足**时沉默地替人做默认决定；它要能识别关键分歧并主动追问，用样例、反例、测试、既有代码风格来恢复真实意图。其次，它要更强于验证，而不只是生成，因为软件工程里最贵的往往不是写出第一版，而是确认这版真的对。再往后，它还必须拥有更强的上下文能力和可控性，知道架构历史、事故经验、权限结构和审计要求，并且能解释、回滚、纳入真实工程流程。说到底，我们真正需要的“更强”，不是更会猜，而是更知道哪里不该猜；不是更像一个高效的平均答案制造机，而是更像一个能保留人类真实意图、把关键决策还给人的协作者。

所以，对“当 Opus 4.6、GPT-5.4 轮番登场之后，我们还需不需要更强的编程模型”这个问题，我的答案是：需要，但前提是我们先把“更强”重新定义清楚。如果更强只是意味着更会补全、更会 debug、更会套用主流架构、更会用行业平均最佳实践把空白补满，那么当 accidental information 已经被吃掉大半之后，这条路的边际收益一定会递减，而且它越成功，软件越容易同质化。

但如果我们说的“更强”，指的是更能识别约束不足、更能把关键决策还给人、更能整合跨项目上下文、更能验证结果、更能在长周期系统演化中保持一致性，那么这就已经不只是模型能力，而是 agent 系统能力，甚至是人与 agent 协作体系的能力。真正仍然需要被继续强化的，未必是一个更强的代码生成模型，而更可能是一个更强的软件生产闭环。
